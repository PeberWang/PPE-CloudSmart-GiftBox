# 自进化循环指南 — PPE云端智能大礼包

> 本文档是大礼包项目五维审查与改进建议的基础参考。每次 cron job 执行审查前，必须先读取本文件和上一次的演进日志（`log/` 目录下最新文件）。

---

## 一、核心理念

本项目的自进化与政策监控项目不同：**不自动执行改进，只生成改进建议，等待用户审批后再动手。**

大礼包项目的特殊性：
- 这是面向真实用户的开源产品，每次改动都可能影响使用者体验
- 项目节奏由铭培主导，AI 只负责观察和建议
- 改进建议需要附带足够的上下文，让铭培能快速决策"做不做、怎么做"

参考原则：
- **先架构后编码** — 没有清晰的设计就不要动手
- **高内聚低耦合** — 每个模块只做好一件事
- **胶水编程** — 最大化复用成熟组件，最小化自写代码
- **面向意图** — 讲需求，不讲步骤

## 二、五维审查体系

### 维度A：模块化与职责分离

**审视目标**：代码结构是否清晰地反映了"资料管理→问答交互→用户界面"的业务流程？

**检查思路**：
1. 画出当前模块依赖关系图（谁调谁）
2. 识别"上帝模块"——如果一个文件超过300行或被5个以上其他文件 import，它可能在承担过多职责
3. 检查新增一门课程的资料需要改动几个文件——理想情况是只改 data 层和 config
4. 检查各功能模块（智能问答、资料检索、用户管理）是否独立——修改一个不应影响其他
5. 检查前端与后端的接口是否清晰、稳定

**改进方向**：抽取公共逻辑、拆分大函数、消除循环依赖

---

### 维度B：鲁棒性与可维护性

**审视目标**：当外部环境变化（LLM API 限流、用户异常输入、文件路径失效）时，系统能否优雅降级而非崩溃？

**检查思路**：
1. 列出所有可能失败的外部调用（HTTP 请求、LLM API、文件 I/O），检查是否有 try/except
2. 错误处理策略是否统一——是集中式还是分散式？哪种更适合？
3. 当用户上传格式不正确的文件时，系统行为是什么？
4. 日志系统——能否从日志中快速定位问题？
5. 重复操作是否安全——同一个文件上传两次会怎样？

**改进方向**：统一异常策略、增强数据校验、完善日志、添加健康检查

---

### 维度C：开源就绪度

**审视目标**：如果今天把代码推到 GitHub 公开仓库，一个陌生学生能在 30 分钟内跑起来吗？

**检查思路**：
1. README.md 是否包含：项目介绍、安装步骤、配置说明、运行方法、贡献指南？
2. 有没有硬编码的路径、密钥、或环境特定的配置？
3. .env.example 和 .gitignore 是否覆盖了敏感文件？
4. 代码风格是否统一：命名规范、注释风格、import 顺序？
5. 有没有 "dead code"——从未被调用的函数、被注释掉的代码块？
6. 依赖管理是否清晰（requirements.txt / pyproject.toml）？

**改进方向**：补全文档、清理 dead code、统一代码风格、消除硬编码

---

### 维度D：可扩展性与用户体验

**审视目标**：如果要新增一门课程、支持新一届学生贡献资料，操作成本有多高？

**检查思路**：
1. 新增一门课程的步骤是什么？写下来，数一数需要几步
2. materials.json 的结构是否自解释？字段名是否见名知义？
3. 用户（贡献资料的学生）使用流程是否简洁？能否一键提交？
4. 问答系统的回答质量如何？有无评估机制？
5. 错误信息是否对用户友好？

**改进方向**：简化资料贡献流程、增强配置验证、改善错误信息、支持批量导入

---

### 维度E：架构理念审查

**审视目标**：这个项目的本质价值主张是什么？当前架构是否支撑了它？

**检查思路**：
1. 用一句话定义这个项目——不只是"资料管理系统"，它解决的核心问题是什么？
2. 当前的数据流（资料导入→索引→检索→问答→反馈）是否有冗余环节？
3. 有没有更好的技术范式可以替代当前实现？（例如：从文件存储改为向量数据库？从关键词匹配改为语义检索？）
4. 项目的长期发展方向是什么？（支持更多学校？更多年级？社区化？）
5. 与同类项目（各高校课程资料共享平台）相比，我们的独特价值在哪里？

**改进方向**：重新定义项目定位、探索新技术方案、规划长期路线图

## 三、执行规范

### 每日执行流程

```
1. 读取 docs/自进化循环指南.md（本文件）
2. 读取 log/ 目录下最新的演进日志
3. 从日志中找到「下次审查维度」字段 → 确定本次审查维度
4. 按该维度的检查思路执行审查
5. 生成改进建议（不执行！）
6. 写入新的演进日志（必须包含「下次审查维度」字段）
7. 将改进建议发送给用户，等待审批
```

### 维度轮换规则

固定顺序：**A → B → C → D → E → A → B → ...**

- 如果日志中找不到「下次审查维度」字段，从 A 开始
- 同一个维度可以连续多轮审查（如果上轮发现问题太多），但日志中必须说明原因

### 与政策监控项目的关键区别

| 维度 | 政策监控项目 | 大礼包项目 |
|------|-------------|-----------|
| 执行方式 | **自动实施改进** | **只生成建议，等用户审批** |
| 改进粒度 | 每次一个实质性改动 | 建议可以是小改动列表 |
| 日志位置 | 项目 log/ | 项目 log/ |
| 汇报方式 | 改进后汇报结果 | 建议发到群聊等用户确认 |
| Git 操作 | 自动 commit + push | **不自动 commit，用户决定后手动** |

### 建议格式

每条改进建议必须包含：
- **问题描述**：具体到文件和行号
- **改进方案**：怎么做
- **预期效果**：改完会怎样
- **风险**：可能引入什么问题
- **优先级**：高/中/低

### 演进日志格式

```markdown
# YYYY-MM-DD-HH 演进日志

## 本次审查维度：[A-E]（维度名称）

## 上次审查回顾
- 上次维度：[X]，发现了什么问题，建议的改进方案是否被采纳

## 本次发现的问题
- [具体到文件和行号]

## 本次改进建议
- [问题 → 方案 → 预期效果 → 风险 → 优先级]

## 等待用户决策
- [列出需要铭培确认的事项]

## 下次审查维度：[下一个字母]（维度名称）
```
